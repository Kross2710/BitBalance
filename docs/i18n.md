# BitBalance i18n — Contributor Guide

BitBalance speaks English by default. The i18n layer lets contributors add new
locales without touching templates — just drop a translation file and register
it. This page is the short version: what to edit, what to expect, what to test.

## How it works (60-second tour)

- **Locale registry** — `include/i18n/locales.php` returns an associative array
  of `code => metadata`. Each code matches a file in the same folder, e.g.
  `vi` ↔ `vi.php`.
- **Translation files** — `include/i18n/<code>.php` returns a flat
  `key => string` array. Missing keys fall back to `en.php`; if even English
  lacks the key, `t()` returns the key itself so missing translations are
  obvious in the UI.
- **Resolution order** at request start (see `resolve_locale()` in
  `include/i18n/i18n.php`):
  1. Logged-in user's `userStatus.language_preference` column.
  2. Cookie `lang` (30 days).
  3. `Accept-Language` header, best-match against the registry.
  4. Fallback: `en`.
- **Helpers** available everywhere after `include/init.php` runs:
  - `t($key, $vars = [])` — HTML-escaped translation. Use this in templates.
  - `t_raw($key, $vars = [])` — raw string. Use only in contexts where the
    caller does its own escaping (JSON, JS template literals with manual
    `escapeHtml()`, attribute values you build yourself).
  - `current_locale()`, `available_locales()`, `html_lang_attr()`.
  - `set_locale($code, $pdo, $userId)` — persists to DB + cookie + session.

## Adding a new language

1. **Pick a code.** Use the lowercase ISO 639-1 tag (`fr`, `ja`) or extend with
   a region tag (`pt-BR`) only if you really need to distinguish — most
   projects don't.
2. **Copy English.** `cp include/i18n/en.php include/i18n/<code>.php`. Don't
   start from scratch — you'll forget keys.
3. **Translate values, keep keys.** Keys are stable contracts with the
   templates. Translate the right-hand side only. Keep `{placeholders}`
   intact and in the right place for your grammar.
4. **Preserve markup.** Some values include intentional HTML (`<br>`,
   `<strong>`, `<span class="gradient-text">…</span>`). They get rendered via
   `t_raw()`, so the tags must stay. Don't add new HTML — open an issue if a
   template needs a structural change.
5. **Register the locale.** Append an entry to `include/i18n/locales.php`:

   ```php
   'fr' => [
       'native'   => 'Français',
       'english'  => 'French',
       'html_lang' => 'fr',
   ],
   ```

   The footer language switcher renders one button per registered locale, in
   array order. `native` is what users see; `english` is the tooltip.
6. **Test the round-trip.** Open the footer switcher, pick the new language,
   confirm it sticks across page loads (cookie + DB), and skim the index +
   dashboard for layout breakage. RTL scripts and very long button labels are
   the usual culprits.

## Conventions

- **Key namespaces:** `area.section.element` (e.g. `dashboard.sidebar.calculator`).
  Common reusable strings live under `common.*`.
- **Placeholders:** `{name}` style — `t('header.xp_tooltip', ['into' => 200, 'ceil' => 500, 'level' => 7])`.
- **HTML in values:** only with `t_raw()` at the call site; never user-supplied
  content.
- **Backend messages:** error/success strings from PHP handlers are currently
  English. Translating them is a follow-up — open a PR per page and call
  `t()` at the point the string is set, not where it's rendered.

## Scope of the current release

Translated UI: `index.php`, `dashboard/*` (overview + intake + calculator +
plan + progress + history + wiki + friends + set-goal + shared partials),
`profile.php`, `ai-coach.php`, plus the shared header, footer, and cookie
banner.

Not yet translated: admin pages, login/signup, forum, products, cart, terms,
about, and the long-form Nutrition Wiki article bodies (translatable but treated
as separate content work). Server-side validation / error messages still emit
English. Contributions welcome — same file conventions apply.

## Migration

The `language_preference` column is added by
`include/migrations/2026_05_29_user_language_preference.sql`. It defaults to
`'en'` so the column is safe to add on a live DB and existing users keep their
current experience until they pick a language.
